# Database Migrations — VazhiAI Backend

This project uses **Alembic** for versioned, production-grade database schema management.

---

## Workflow: Everyday Commands

### 1. Check current migration state
```bash
alembic current
```

### 2. View pending migrations (not yet applied)
```bash
alembic history --verbose
```

### 3. When you change a model in `db_models.py`, generate a new migration
```bash
alembic revision --autogenerate -m "describe_your_change"
```
> Always review the generated file in `migrations/versions/` before applying it.

### 4. Apply all pending migrations to the database
```bash
alembic upgrade head
```

### 5. Roll back to the previous migration version
```bash
alembic downgrade -1
```

### 6. Roll back to a specific revision
```bash
alembic downgrade <revision_id>
```

---

## Switching to AWS RDS

When you migrate to AWS RDS, only one thing changes:

Update the `DATABASE_URL` in your environment (`.env` or hosting dashboard):
```
DATABASE_URL=postgresql://db_user:db_password@your-rds-endpoint.rds.amazonaws.com:5432/VazhiAI_db?sslmode=require
```

Then run migrations against the new database:
```bash
alembic upgrade head
```

Alembic will detect it's a fresh database and apply all migrations from scratch.

---

## Files

| File | Purpose |
|---|---|
| `alembic.ini` | Alembic configuration (timestamped filenames, no hardcoded URLs) |
| `migrations/env.py` | Reads DATABASE_URL from `.env`, imports all models for autogenerate |
| `migrations/versions/` | All versioned migration scripts (never edit manually) |

---

## Important Rules

- **Never delete** files from `migrations/versions/` — they are the history of your schema.
- **Always review** autogenerated migration files before running `upgrade head`.
- **Never hardcode** database credentials in `alembic.ini` or `env.py`.
